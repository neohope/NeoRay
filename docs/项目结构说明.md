# NeoRay 项目结构说明

> 本文档由 Codex 根据当前仓库实际结构生成，用于说明各目录职责、设计问题与重构建议。

## 总览

NeoRay 是一个 AI Agent 框架，由两部分组成：

- **Go 后端** (`neoray/`)：核心代码库（`internal/` 下约 1.5 万行），承载全部业务逻辑。
- **Flutter 前端** (`neoray_app/`)：轻量客户端，负责聊天 UI 与配置界面。

另有文档 (`docs/`)、构建脚本 (`scripts/`)、设计文件 (`design/`) 与被 gitignore 的外部参考代码 (`reference/`)。

## 目录职责

### 根目录
- `neoray/` — Go 后端。入口 `neoray/cmd/server/main.go`，逻辑集中在 `neoray/internal/`。
- `neoray_app/` — Flutter 客户端，真实代码在 `neoray_app/lib`。
- `docs/` — 架构、配置、路线图等文档（部分已过时）。
- `design/` — 单个 `.pen` 设计文件。
- `scripts/` — 根级跨平台构建编排脚本（调用各子项目脚本）。
- `reference/` — 外部参考项目（`oapi-sdk-go`、`neonanobot`），已 gitignore，仅供移植参考。
- `test/` — 工作区根部的测试目录，当前仅有空的 `internal/command/` 子目录，无实际文件。

### 后端 `neoray/internal/`
- `agent/` — 核心循环与回合状态机、上下文/Token/Trace 管理（`agent.go`、`loop.go`）。
- `provider/` — LLM 抽象层（Anthropic、OpenAI 兼容、fallback）。
- `tools/` — 工具系统（shell、filesystem、patch、web、search、cron、memory 等）。
- `memory/` — 长期记忆：git 存储、dream / consolidator / autocompact。
- `channel/` — 聊天平台接入；`feishu.go` 单文件 2100+ 行。
- `api/` — HTTP + WebSocket 服务。
- `bus/` — 异步消息总线与事件发布订阅。
- `session/` — 会话实体、文件存储、目标状态管理。
- `config/` — 强类型配置与加载器。
- `security/` — 工作区限制、沙箱、网络策略。
- `cron/` — 定时任务调度。
- `subagent/` — 子代理管理与生成工具。
- `command/` — 内置命令与路由。
- `skills/` — 从磁盘加载 Markdown 技能。
- `templates/` — 从磁盘加载 Markdown 提示词模板（单例、多路径探测）。
- `logger/` — 基于 zap 的日志。
- `tui/` — Bubble Tea 终端界面。

### 后端运行时资源
- `neoray/skills/` — 内置技能的 `SKILL.md`。
- `neoray/templates/` — 提示词/身份模板（`.md`）与配置模板。

### 前端 `neoray_app/lib/`
- `models/` — 数据模型（freezed / json 生成）。
- `pages/` — 聊天页、配置页。
- `providers/` — Riverpod 状态。
- `services/` — API 与 WebSocket 客户端。
- `widgets/`、`theme/`、`utils/` — UI 组件、主题、工具。

## 设计不合理 / 应重构

- **`cmd/server/main.go` 臃肿**：约 200 行 cron 的 `map[string]any` 转换适配器（`parseSchedule`、`jobToMap` 等）写在 `main` 包，应下沉到 `internal/cron`；另有 55 处 `fmt.Println` 调试输出，应改为 `logger` 或删除。
- **`channel/feishu.go`（2100+ 行）巨型单文件**：应拆分为传输、事件解析、消息格式化、流式输出等文件。
- **`agent` 包过大**（`agent.go` 948 行 + `loop.go` 851 行）：`Agent` 结构体字段超过 20 个、大量 `With…` 选项，近似“上帝对象”，建议拆解回合状态机与依赖装配。
- **模板/技能靠运行时探测磁盘路径**：`templates.go` 探测 6 个候选目录，全项目未用 `go:embed`，导致二进制强依赖同级 `templates/`、`skills/` 目录，分发脆弱。建议改用 `go:embed`。
- **配置文件漂移**：`neoray/config.toml`、`neoray/config.toml.example`、`neoray/templates/config.toml` 三份近似配置行数不一（128 vs 158），应收敛为单一来源。
- **测试目录布局混乱**：`internal/subagent/subagent_test.go` 位置正确，但 `neoray/test/internal/command`、`neoray/test/internal/skills` 又建了平行测试树，不符合 Go 惯例。

## 明显缺陷 / 不足

- **测试覆盖薄弱**：约 1.5 万行后端仅 3 个测试文件，`agent`、`provider`、`memory`、`channel`、`api` 等核心无测试。
- **WebSocket 安全隐患**：`api/server.go` 的 `CheckOrigin` 恒返回 `true`（允许所有来源），代码注释自认“生产环境应限制”。
- **调试输出与信息泄漏**：`main.go` 散落 `fmt.Println`，包括打印 provider API key 长度，应改走 logger。
- **文档与实现不符**：`README.md` 仍称前端为 `neorayui/` 并引用不存在的 `config/` 目录（实际为 `neoray_app/`）；`BUILD.md` 提到 `config.yaml` 而代码用 `config.toml`；README 声称“Phase 2 进行中”，实际进度远超。

## 无价值 / 可剔除

- **`internal/memory/integration_example.go`**：整文件是注释形式的用法示例，应移入文档，不应作为编译源文件。
- **根 `test/` 目录**：仅有空 `internal/command/` 子目录，属死脚手架。
- **重复 `.exe` 二进制**：磁盘上有 `server.exe`、`neoray/neoray.exe`、`neoray/server.exe`（各约 17MB），已 gitignore，属陈旧冗余，应清理并交由构建脚本产出。
- **重复配置副本**：按上文收敛。
- **`reference/`**：体量大（完整 `oapi-sdk-go` SDK + `neonanobot`），已 gitignore，若仅一次性移植参考可从工作副本移除以降噪。
- **`design/neoray_v1.pen`**：单个私有设计文件，可保留但无代码价值。
